# Decision Scenarios

Four filled framework examples for common decision types. Use these as structural patterns, not copy-paste templates.

---

## Process Change Decision

```
Decision: Change [current process] to [new approach]

Current State Pain:
- [Problem 1]
- [Problem 2]

Options:
A) Major overhaul - [description]
B) Incremental improvement - [description]
C) Status quo + workarounds - [description]

Analysis: [Comparison]

Recommendation: [B - incremental]

Rationale: Addresses core pain ([Problem 1]) without requiring [major
change that's risky]. Can iterate based on results. Aligns with "crawl
before run" approach.

Risks: [May not fully solve Problem 2, will need fast-follow]
Mitigation: [Plan for Phase 2 after seeing Phase 1 adoption]
```

---

## Stakeholder Conflict Decision

```
Decision: How to resolve [conflict] between [Party A] and [Party B]

Context:
- Party A wants: [X]
- Party B wants: [Y]
- Underlying issue: [Root cause]

Options:
A) Side with Party A - [implications]
B) Side with Party B - [implications]
C) Compromise approach - [middle ground]
D) Escalate to leadership - [when this makes sense]

Analysis: [Comparison]

Recommendation: [C - compromise]

Rationale: Addresses Party A's core need ([X]) while respecting Party B's
constraint ([Y]). Maintains both relationships. Doesn't require executive
escalation.

Implementation: [How to position with each party]
```

## Resource Allocation Decision

```
Decision: Allocate [resource/budget] to [Project A] vs [Project B]

Context:
- Total available: [Amount]
- Project A need: [Amount] - Value: [Impact]
- Project B need: [Amount] - Value: [Impact]

Options:
A) Fund Project A fully - [implications]
B) Fund Project B fully - [implications]
C) Split funding - [implications]
D) Defer one project - [implications]

Analysis: [Comparison based on ROI, strategic alignment, urgency]

Recommendation: [A - Project A]

Rationale: Higher strategic value ([specific]), shorter timeline to impact
([metric]), addresses leadership priority ([which one]).

Communication Plan:
- Project A: [How to message approval]
- Project B: [How to message deferral without burning relationship]
```

## Technology/Vendor Selection

```
Decision: Select [Tool/Vendor] for [use case]

Requirements:
- Must-have: [Feature 1], [Feature 2], [Feature 3]
- Should-have: [Feature 4], [Feature 5]
- Nice-to-have: [Feature 6]

Options:
A) Vendor A - [capabilities]
B) Vendor B - [capabilities]
C) Build in-house - [implications]

Analysis:
[Comparison matrix with scoring]

Recommendation: [Vendor A]

Rationale: Meets all must-haves, stronger on [key differentiator], better
long-term fit for [strategic direction].

Risks & Mitigation:
- Risk: [Vendor lock-in] - Mitigation: [Contract terms, exit strategy]
- Risk: [Integration complexity] - Mitigation: [Pilot first, phased rollout]

Implementation Plan:
- Phase 1: Pilot with [small group] - [Timeline]
- Phase 2: Evaluate and adjust - [Timeline]
- Phase 3: Full rollout - [Timeline]
```
